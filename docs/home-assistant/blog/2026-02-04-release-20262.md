## 查看家庭的新方式

随着 Home 仪表盘成为官方默认标准，它现在会显示为“概述”，并在所有新实例中取代旧版“概述”。如果你是长期用户且从未自定义过默认视图，我们会向你推荐切换；否则，你也可以随时前往 [**设置** > **仪表盘**](https://my.home-assistant.io/redirect/dashboards/) 手动体验。

<img src='/home-assistant/images/blog/2026-02/new-overview.png' alt='Screenshot of the new 概述 page' class='no-shadow' />

如果你喜欢把旧版概述当作自定义仪表盘的起点，也完全没问题。前往 [**设置** > **仪表盘**](https://my.home-assistant.io/redirect/dashboards/)，选择 **Create**，再选用 **概述 (legacy)** 模板即可。

### 一眼查看已发现设备

<img src='/home-assistant/images/blog/2026-02/discovered-devices.png' alt='Screenshot of the modal view to add discovered 设备 from 概述' class='no-shadow' />

看看 **For You** 分区里的新卡片！它会即时显示 Home Assistant 新发现的设备，让你可以当场添加，或直接跳转到设备管理页面，不用再层层翻菜单。

### 轻松完成区域分配

<img src='/home-assistant/images/blog/2026-02/assign-devices-page.png' alt='Screenshot of assigning 设备 to areas from 设备 page in 概述' class='no-shadow' />

在上一次发布中，我们在 Home 仪表盘里新增了一个专门的 **设备** 区域，用于收纳所有尚未分配的设备。现在这个区域会提供快捷提示，帮助你把设备快速归类到正确房间，让你的设置几乎不费力就能保持整洁。

### 更快的区域编辑

想替换某个区域的温度传感器？现在区域页面在 **Edit** 按钮中提供了快捷入口。你可以直接跳到该区域配置页，几秒内更新温湿度等主传感器。

我们还整理了界面细节，移除了别扭的留白，并修复了一些返回箭头问题。现在在各级子菜单中导航会更顺滑，也更符合你的预期。

### 体验与视觉升级

**默认主题更现代**：我们移除了旧版蓝色顶栏，改用与 **设置** 页面一致、简洁统一的主题。这种更少干扰的设计能让卡片和数据真正成为主角。

**每位用户都可个性化主题**：主题入口位置调整了！你现在可以直接在 [**User profile**](https://my.home-assistant.io/redirect/profile/) 中查找并切换喜欢的外观，让你在任何已登录的设备上都能更轻松地使用熟悉主题。

## 设备数据库：我们需要你的帮助！

在购买智能家居设备前，想找到可靠信息并不容易。这正是我们建设 **Open Home Foundation 设备数据库** 的原因：打造一个由社区共同驱动、基于真实世界数据的资源，帮助你做出更明智的选择。

我们已经与早期贡献者一起完成了基础建设，而[当前成果已经非常亮眼](https://openhomefoundation.grafana.net/public-dashboard/1cb22c82e90c4f64afb366c6125a8489)：选择加入并分享匿名数据的 Home Assistant 用户，已提交超过 **10,000 台独立设备**，覆盖 **260+ 个集成**。

<img src='/home-assistant/images/blog/2026-02/device-database-statistics.png' alt='Screenshot of the publicly available statistics 仪表盘 for the open home foundation 设备 database.' class='no-shadow' />

### 分享你的设备，帮助我们

由于项目仍处于早期阶段，设备数据库目前位于 [**Home Assistant Labs**](https://my.home-assistant.io/redirect/labs/)，你可以在那里选择加入，分享家中设备的匿名信息。

<img src='/home-assistant/images/blog/2026-02/sharing-your-devices-with-ohf-device-database.png' alt='Screenshot of the 设备 analytics sharing option in Home Assistant Analytics.' class='no-shadow' />

我们还在 [**Home Assistant Analytics**](https://my.home-assistant.io/redirect/analytics/) 中新增了 **设备 analytics** 分区；当你在 Home Assistant Labs 启用后，它就会出现。你随时可以选择退出。

<img src='/home-assistant/images/blog/2026-02/labs-device-database-analytics.png' alt='Screenshot of the 设备 analytics section in Home Assistant Labs.' class='no-shadow' />

**隐私是我们的根基。** 我们不会收集任何个人数据。只有在用户主动选择加入后，才会共享汇总且匿名化的设备信息，在保护隐私的同时提供有价值的洞察。你可以在 Analytics 页面右上角通过 **Preview 设备 analytics** 选项预览将要发送的数据。完整细节请阅读我们的 [Data Use Statement](https://www.openhomefoundation.org/devices-database-data-use-statement/)。

## Add-ons 现在改名为 Apps

从这次发布开始，**add-ons 正式改名为 apps**！🎉

<img src='/home-assistant/images/blog/2026-02/addons-are-now-apps.png' alt='Screenshot showing the 设置 menu, that now contains the Apps items instead of Add-ons (as it was called previously)' class='no-shadow' />

你可能会问：为什么要改名？答案很简单——让 Home Assistant 对所有人，尤其是新用户，更容易理解和上手。

当你第一次打开 Home Assistant 时，会看到两个听起来很像的分区：“Add-ons”和“集成”。它们都像是在“添加某些东西来扩展 Home Assistant”，但实际用途完全不同。对老用户来说这种差异早已习惯，但我们持续看到新用户被混淆：需要集成时去安装 add-ons，或反过来。

这正是改名的价值：使用大家已经熟悉的术语。大多数人都知道什么是“app”。你打开手机应用商店，选一个 app，然后安装。电视有应用商店，NAS 有 apps，甚至现在有些冰箱都有 apps。这是所有人都能理解的概念。现在，同样的心智模型也适用于 Home Assistant：

- **Apps** 是在 Home Assistant 旁边独立 _运行_ 的应用程序。
- **集成** 是把 Home Assistant 与你的设备和服务 _连接起来_ 的桥梁。

Apps 是由 Home Assistant Operating System 管理的独立软件，与 Home Assistant 本体并行运行。它们可以是代码编辑器、媒体服务器、MQTT broker 或数据库工具。有些 apps 还会与集成配合使用：例如 Mosquitto MQTT broker app 提供服务，而 MQTT 集成负责把 Home Assistant 连接到该服务。

现有文档、社区帖子和教程在一段时间内仍会继续使用“add-ons”这一称呼。搜索引擎和 AI 助手也需要时间跟进。我们已经设置了重定向，确保你搜索“add-ons”时仍能到达正确位置。

感谢所有参与[社区讨论](https://community.home-assistant.io/t/why-im-proposing-we-rename-add-ons-to-apps-and-why-it-matters-for-newcomers/945712)和[架构提案](https://github.com/home-assistant/architecture/discussions/1287)的朋友。无论你是支持、反对，还是持中间观点，你的反馈都非常宝贵。

### 更快、更顺滑的 Apps 面板

除了改名，本次发布还对 Apps 面板（此前叫 Add-ons 面板）底层做了大规模重构。过去这个面板由独立进程（Supervisor）提供，现在它已经完全整合进 Home Assistant 前端。

视觉上你可能感觉变化不大，但使用起来会 _明显_ 更快、更顺滑。更重要的是，这项调整让后续 Apps 相关开发容易了很多，也为未来更多改进铺平了道路。

## 面向用途的触发器与条件进展

在 [Home Assistant 2025.12](/home-assistant/blog/2025/12/03/release-202512/#purpose-specific-触发器-and-条件) 中，我们引入了面向用途的触发器与条件。创建自动化时，你不必再从技术性状态变化去思考，而是可以直接选择“当灯光打开时”或“当温控正在制热时”这类自然描述。在 [Home Assistant 2026.1](/home-assistant/blog/2026/01/07/release-20261/#purpose-specific-触发器-and-条件-progress) 中，我们新增了更多触发器，并为条件打好了基础。

这项功能仍在 [**Home Assistant Labs**](https://my.home-assistant.io/redirect/labs/) 中持续打磨，但我们每次发布都在扩展它。本次发布不仅带来了新的触发器，还首次加入了一整套面向用途的条件！

### 新增触发器

以下是本次发布新增的触发器：

- **Calendar** 触发器会在日历事件开始或结束时触发。
- **Person** 触发器现在支持在人物到家或离家时触发。
- **吸尘器** 触发器会在扫地机器人返回底座时触发。

### 新增条件

面向用途的条件也在持续扩展！在上一次发布中，我们先为灯光引入了首批面向用途条件；这次则把这套能力扩展到了更多实体类型。

<img src='/home-assistant/images/blog/2026-02/new-conditions-for-automations.png' alt='Screenshot showing the newly available media player 条件: check if a media player is on, off, playing, paused, or not playing.' class='no-shadow' />

和触发器一样，条件现在也能让你用更自然的方式表达意图。你不必再判断“某实体状态是否等于某值”，而是可以直接写成“如果温控正在制热”或“如果门锁已上锁”。

目前可用的面向用途条件包括：

- **报警控制面板** 条件：判断报警是否已布防（在家、离家、夜间或度假）、已撤防或已触发。
- **Assist satellite** 条件：判断语音卫星当前是空闲、监听中、处理中还是响应中。
- **温控** 条件：判断温控设备是开启、关闭、制热、制冷或除湿状态。
- **设备追踪器** 条件：判断设备是否在家。
- **风扇** 条件：判断风扇是否开启。
- **加湿器** 条件：判断加湿器是开启、关闭、加湿中或除湿中。
- **Lawn mower** 条件：判断割草机是正在作业、已回充、已暂停、正在返回或发生错误。
- **门锁** 条件：判断门锁是否上锁、解锁、开启或卡住。
- **Media player** 条件：判断媒体播放器是开启、关闭、播放中、暂停中或未播放。
- **Person** 条件：判断人物是否在家。
- **Siren** 条件：判断警笛是否开启。
- **开关** 条件：判断开关是否开启。
- **吸尘器** 条件：判断吸尘器是清扫中、已回充、已暂停、正在返回或发生错误。

前往 [**设置** > **System** > **Labs**](https://my.home-assistant.io/redirect/labs/) 启用面向用途的触发器与条件，马上试试吧！

## 全新卡片：distribution 卡片

认识一下 **distribution 卡片**：这是一张全新的仪表盘卡片，用于可视化多个实体之间的数值分布。它会以按比例横向条形图搭配交互式图例展示数据，让你一眼看清电力、存储或其他可度量资源分别流向哪里。

<img src='/home-assistant/images/blog/2026-02/distribution-card.png' alt='Screenshot of two distribution 卡片 on a desktop, providing new insights into your data.' class='no-shadow' />

这张卡片支持完整交互：点击图例项可隐藏或显示实体（百分比会动态重算），点击条形片段可打开该实体的更多信息对话框。当实体较多时，图例会先展示前几项，并提供 **More** 按钮展开其余项。

distribution 卡片会智能校验可组合对象。它会确认所有实体属于相同 domain 和设备类别，避免你把功率传感器和电池传感器混在一起。它也能优雅处理相关单位：比如瓦特和千瓦混用也没问题。

你可以这样使用它：

- **功率监控**：查看当前哪些回路或电器最耗电。
- **存储使用情况**：可视化磁盘或文件夹之间的存储分布。
- **任意比例型数据**：比较任意一组同单位实体。

## Quick search：通往一切的最快方式

我们持续在让 Home Assistant 的访问与检索变得更简单。quick bar 现已彻底重设计，并正式更名为 **Quick search**。你可以把它当作整个 Home Assistant 的指挥中心：跳转页面、执行命令、查找实体、设备或区域，全都通过一个统一搜索入口完成。

<img src='/home-assistant/images/blog/2026-02/quick-search.png' alt='Screenshot of the Quick search interface showing category filters and search results.' class='no-shadow' />

无论在哪个页面，你都可以通过 macOS 的 <kbd>⌘</kbd> + <kbd>K</kbd>，或 Windows/Linux 的 <kbd>Ctrl</kbd> + <kbd>K</kbd> 打开 Quick search。新设计在顶部提供了分类筛选：**Navigate**、**Commands**、**实体**、**设备** 和 **Areas**。你可以先选分类快速缩小结果，也可以直接输入跨分类搜索。

完整键盘导航让 Quick search 成为重度用户的好伙伴：方向键移动结果，<kbd>Enter</kbd> 选中，<kbd>Esc</kbd> 关闭。在移动端，你还可以把 Quick search 绑定到手势，实现一触即达。

### 你熟悉的快捷键仍然可用

如果你之前一直在用旧 quick bar 的单键快捷方式，它们依旧可用！区别在于：现在会直接打开并预选对应筛选器的 Quick search：

- <kbd>e</kbd>：以 **实体** 筛选打开 Quick search
- <kbd>d</kbd>：以 **设备** 筛选打开 Quick search
- <kbd>c</kbd>：以 **Commands** 筛选打开 Quick search
- <kbd>a</kbd>：仍然直接打开 **Assist**
- <kbd>m</kbd>：仍然为当前页面创建 **My link**（虽与搜索无关，但依旧实用！😉）

这意味着你的肌肉记忆依然有效，同时又能获得所有新能力。

## 需要帮助？加入社区

Home Assistant 拥有一个非常棒的用户社区，大家都很乐意互相帮助。所以，快来加入我们吧！

我们非常活跃的 [Discord 聊天服务器](/home-assistant/join-chat) 是一个很好的去处，也别忘了加入我们的精彩 [论坛](https://community.home-assistant.io/)。

发现缺陷或遇到问题？请在我们的 [问题跟踪器](https://github.com/home-assistant/core/issues) 中反馈，以便我们修复！你也可以查看我们的 [帮助页面](/home-assistant/help)，了解还有哪些地方可以寻求帮助。

如果你更喜欢电子邮件，也可以[订阅 Open Home Foundation 新闻通讯](/home-assistant/newsletter)，把最新功能动态、社区新闻，以及支持 Open Home 的其他项目资讯直接发送到你的收件箱。

## 不向后兼容的变更

我们始终尽力避免对现有功能做出可能意外影响你 Home Assistant 安装的改动，但有时这确实无法避免。

我们始终会把这些变化记录清楚，尽可能让过渡更顺利。本次发布包含以下不向后兼容的变更：

传感器组的行为已经发生变化：

- 如果组内所有成员都处于 unavailable 或缺失状态（也就是不在状态机中），那么传感器组现在也会显示为 unavailable。
- 当该组不被视为 unavailable，且配置变量 `ignore_non_numeric` 设置为 `False`（默认值）时，只有在所有成员都存在于状态机中并且具有数值状态的情况下，组状态才会按配置的类型进行计算；否则组状态将为 unknown。

([@emontnemery] - [#152167]) ([group 文档])

[@emontnemery]: https://github.com/emontnemery
[#152167]: https://github.com/home-assistant/core/pull/152167
[group 文档]: /integrations/group/

仅适用于自托管 Sentry 用户：由于 SDK 使用了 `/envelope` API 端点，本次升级要求 Sentry 服务器版本至少为 20.6.0（发布于 2020 年 6 月）。运行更旧版本自托管 Sentry 的用户，必须先升级服务器，再更新 Home Assistant。

使用 sentry.io 的 Home Assistant 用户不受影响。

([@vaind] - [#159415]) ([sentry 文档])

[@vaind]: https://github.com/vaind
[#159415]: https://github.com/home-assistant/core/pull/159415
[sentry 文档]: /integrations/sentry/

以下传感器已被移除，因为 Tractive API 已不再支持它们：

- `activity`
- `calories burned`
- `sleep`

如果你在自动化或脚本中使用了这些实体，则必须更新它们。

([@bieniu] - [#160089]) ([tractive 文档])

[@bieniu]: https://github.com/bieniu
[#160089]: https://github.com/home-assistant/core/pull/160089
[tractive 文档]: /integrations/tractive/

重复的 HVACMode 现已转换为 preset。你可能需要在自动化或脚本中，把服务调用从 `set_hvac_mode` 调整为 `set_preset_mode`。

([@epenet] - [#160918]) ([tuya 文档])

[@epenet]: https://github.com/epenet
[#160918]: https://github.com/home-assistant/core/pull/160918
[tuya 文档]: /integrations/tuya/

`advanced_sleep` 预设模式现已被 `sleep` 取代。如果你在自动化或脚本中仍使用 `advanced_sleep`，则必须将其更新为 `sleep`。
