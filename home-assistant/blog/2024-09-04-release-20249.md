## Sections 变大了

为了给用户带来更多仪表盘定制和组织选项，我们在 Sections 视图中加入了一些新能力，这也真正释放了[可调整卡片尺寸](/home-assistant/blog/2024/07/03/release-20247/#resize-the-cards-on-your-dashboard)的潜力。这些强大的自定义功能可能会影响[卡片或分区布局的可预测性](/home-assistant/blog/2024/03/04/dashboard-chapter-1/#drag-and-drop-rearrangement-of-cards-and-sections)，尤其是在桌面与移动视图之间切换时，因此我们发现它更适合用在较大的卡片上。和往常一样，怎么使用，完全取决于你。

### 更宽的部分

部分可以扩展以跨越多个部分。例如，现在您可以让安全网络头源跨越两个部分或拥有一个巨大的三部分宽地图。部分和部分将继续遵循 [Z 网格规则](/home-assistant/blog/2024/03/04/dashboard-chapter-1/#drag-and-drop-rearrangement-of-卡片-and-sections)，如果新的较宽部分不适合屏幕的特定部分，这可能会导致空白。如果您不喜欢各部分之间的空白，您可以选择\_密集\_布局。密集部分布局选项将尝试遵循 Z 网格规则，但会尽可能填充空白。这可能会影响各部分的顺序和可预测性，但会最大化屏幕空间。

当切换到较小的屏幕时，这些较大的部分将恢复为一个部分宽（如果这会导致某些大的反馈问题，我们建议使用我们的设备[可见性设置](/home-assistant/blog/2024/06/05/release-20246/#control-visibility-of-卡片)）。

<img src="/home-assistant/images/blog/2024-09/section-settings.png" alt="分区设置的截图">

### 全宽卡片

更大的分区也意味着可以容纳更大的卡片。不只是单纯拉伸布局，现在卡片还有一个新选项，可以让它始终占满整个分区宽度。

### 徽章

谈到仪表盘，我们在上次发布中[介绍](/home-assistant/blog/2024/08/07/release-20248/#new-badges-for-your-仪表盘) 的新样式徽章现在更加可配置。

您可以自己决定要显示的元素，而不是使用显示模式。例如，这允许您隐藏徽章的图标，并仅显示您想要在徽章中显示的设备的状态。

## 追踪你未追踪的能量

四月份，我们在能源仪表盘中引入了[单个设备的新能源图表](/home-assistant/blog/2024/03/06/release-20243/#new-energy-graph-for-individual-devices)。这使您可以将所有能源监控聚合到一个图表中，从而使您可以快速查看家里最大的耗能设备。非常感谢 [@karwosts](https://github.com/karwosts)，该图的原始创建者，现已更新。

<img src="/home-assistant/images/blog/2024-09/untracked-energy.png" alt="单个设备能源图表的截图">

我们不可能对每个设备都进行能源跟踪，如果您有整个家庭的能源跟踪（例如智能电表或 [DIY 的东西](/home-assistant/home-energy-management)），更新的图表将计算差异并显示您家的未跟踪使用情况。因此，跳上你的[能量仪表盘](https://my.home-assistant.io/redirect/energy/)并开始解决你的幻象能量负载。

## LLM 代理选项增加

当我们在语音助手中添加使用[大型语言模型(LLM)](/home-assistant/blog/2024/06/05/release-20246/#dipping-our-toes-in-the-world-of-ai-using-llms)的功能时，开始的对话代理数量有限。其中包括基于云的[OpenAI](/home-assistant/components/openai_conversation/)和[Google AI](/home-assistant/components/google_generative_ai_conversation/)集成，以及本地运行的[Ollama](/home-assistant/integrations/ollama/index.md)集成。现在，这三个人都可以控制您家中的设备。

[@Shulyaka](https://github.com/Shulyaka) 现在通过 [Anthropic](/home-assistant/integrations/anthropic.md) 添加了另一个云选项 - 再次感谢您为我们在 Home Assistant 中尝试 AI 提供了另一个选项。设置它可能与其他会话代理对接不同，因此请查看[文档](/home-assistant/integrations/anthropic.md)。我们已经更新了我们的 [LLM 排行榜](https://github.com/allenporter/home-assistant-datasets/tree/main/reports) 以包括 Anthropic。 Claude 3 Haiku 和 Claude 3 Sonnet 在我们的基准测试中得分都非常高，现在名列前茅。 根据基准测试的结果，我们将更便宜的 Claude 3 Haiku 设置为 Anthropic 集成的默认模型。

[@allenporter]添加了配置[Ollama](/home-assistant/integrations/ollama.md)集成的上下文窗口大小的功能。默认大小已增加，性能提高了20%！

## Aqara 加入 与 Home Assistant 配合使用

<img src="/home-assistant/images/blog/2024-09-aqara-wwha/art.jpg" alt="Aqara 通过 Matter 加入 Works with Home Assistant">

我们[最近宣布](/home-assistant/blog/2024/09/03/aqara-joins-works-with-home-assistant/) 有四款 Aqara 设备加入我们的[与 Home Assistant 配合使用](https://partner.home-assistant.io/) 计划。真正令人兴奋的是，这些都是 Matter 设备，Aqara 将成为第一个使用我们的“Works via Matter with Home Assistant”徽章的合作伙伴，提供了一些与 Home Assistant 令人增强的 Matter 集成一起使用的优秀选项。

更重要的是，这将是自[我们最近更新该计划](/home-assistant/blog/2024/08/08/works-with-home-assistant-becomes-part-ohf/)以来第一个加入的合作伙伴，这意味着我们自己测试了逐个设备认证。这使您可以轻松了解哪些内容为您带来Home Assistant可以带来的最佳体验。

## 开源合规

我们投入了大量的工作来确保 Home Assistant 从上到下都是真正的开源。这意味着确保我们的集成所依赖的每个库都使用合规的许可证。我们现在有超过 2,800 个集成，所以你可以想象，这不是一个小任务。我们与代码所有者和库开发人员合作，煞费苦心地检查每个库现在都使用 [OSI 批准的许可证](https://opensource.org/license)。这可能看起来像一个小程序性的事情，但对于保持集成和 Home Assistant 的功能在遥远的未来至关重要。如果您不知道，库是许多设备和服务集成所依赖的功能，调用通过 [工艺品](https://开发者.home-assistant.io/docs/api_lib_index/) 位于 Home Assistant 项目之外的附加协议代码特定。如果个人或公司无法再维护库，适当的开源许可证允许其他人拾取并继续使用。

我们尽力避免对现有功能进行更改，这可能会导致
意外地影响您的 Home Assistant 安装。不幸的是，有时，
这是不可避免的。

对你来说是可能的。此版本具有以下向后不兼容的更改：仅当您将记录器集成与 MySQL 或使用时，此通知才适用
PostgreSQL 数据库。如果您使用默认的 SQLite 数据库，
你可以忽略它。

上次发布包括数据库升级/迁移，增加了
我们可以为标识符字段存储的数字。

尽管标记为成功，但并非所有迁移都成功。这个
发布将再次进行迁移，但会捕获不成功的迁移。

这可能会导致数据库升级比平时花费更长的时间，具体取决于
您的数据库的大小。

请耐心等待升级过程完成。请勿中断
升级过程，因为这可能会导致数据库损坏。
