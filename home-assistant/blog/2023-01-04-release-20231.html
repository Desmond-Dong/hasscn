    #语音之年！

    You should have heard about it by now; 2023 is Home Assistant's year of Voice! It is our goal for 2023 to let users control Home Assistant in their own language. As covered by The Verge, ArsTechnica, and many others.

    The work on voice has already started by collecting intents in all the different languages we want to support. Intents describe the user's intention with what they've said (or written), so Home Assistant can understand what to do, like the 动作 to take and which 设备 are involved.

    All the different possible intent sentences are now being collected in the Intents for Home Assistant GitHub repository, and we are looking for contributions! So, feel free to jump in and help out!

    An 概述 of the current status of all intents and languages can be found on this page. The page gives insight into the parts we need help with.

    Lastly, we are also looking for language leaders! Each language is maintained by one or more language leaders. Language leaders are responsible for reviewing their language's contributions and making sure they are grammatically correct.

    Anyone can apply to become one. If you want to apply to be a language leader, join us in #devs_voice on Discord or open a discussion topic.

    #实体 aliases for voice assistants

    The first little thing related to the voice project found in Home Assistant: Aliases for 实体 names are now configurable via the UI.

    Currently, configured aliases are used by Google Assistant and, of course, can later be used by our own voice assistant.

    These aliases are helpful in case you call the same 设备 by multiple names or when using a voice assistant in multiple languages at the same time.

    The new aliases feature can be found on each 实体's more information dialog in the 设置 tab in the advanced section. If you use Home Assistant Cloud, you can also manage your 实体 aliases (and the 设备 to expose to Google Assistant) in your Home Assistant Cloud 设置.

    Multi-language can be an important factor when using Google Assistant, as those can be configured to support two languages at the same time. For example, English and Dutch simultaneously. Aliases allow you to provide names for translations as an alias. For instance, I have added "Woonkamer stofzuiger" (Dutch) as an alias to my living room 吸尘器.

    It was previously possible to 配置 Google Assistant-specific aliases via YAML as well, and this remains working. You can use either YAML or easily manage them via the UI.

    #Matter 进展

    Last December 发布, we introduced Matter support, and progress has been made since then. The code base has seen many fixes and improvements, and much effort has been put into adding more automated tests and quality assurance.

    There are also notable improvements; commissioning Matter 设备 have been improved (and fixed). So, adding your first Matter 设备 to Home Assistant should be a much better experience now.

    Lastly, the Home Assistant Companion app for Android with support for pairing Matter 设备 has been released! Thanks to the fantastic work by @jpelgrom on the Android app, the commissioning experience is now even better! Here is a quick video showing how that works:

    This does require your Android 设备 to be updated with Matter support which Google is rolling out within the Google Home app in recent Android versions.

    Support for pairing Matter 设备 using our iOS Companion app is being worked on.

    #扩展日历支持

    The last 发布 introduced the local calendar, which included the capability to manage calendar events from Home Assistant. These features have now been added to the Google Calendar 集成 as well!

    Meaning you can now create a new event in your Google Calendar, straight from within Home Assistant. Neat!

    Furthermore, the local calendar has been extended to provide editing support for existing events 🤘

    Lastly, support for monthly variations in the recurrence rules has been added. This includes repetitions like: on the 20th day or the first Wednesday of the month. The latter, of course, is helpful to put the Home Assistant 发布 day on your calendar 😉

    #Translation improvements for 实体

    The support for translations for 实体 has been improved, especially those of 实体 属性! A relatively small change that is improving the UI experience quite a bit.

    Some 实体 provide additional properties, for example, 温控 实体 provide presets or 风扇 modes. 集成 can now offer translations for those 属性 values.

    Additionally, textual 传感器 实体 can now provide a list of possible 状态 they can be in, for which, of course, translations can be provided for now too.

    In the above screenshot, on the left, you can see how the presets of my thermostat now show up nicely in a human-readable format (instead of, for example, "anti_frost" in previous releases).

    On the right, the Moon 传感器 used in an 自动化 触发器, Home Assistant can now suggest 状态 you can use in those cases. Those suggestions will also be translated into the language you are using.

    #不向后兼容的变更

    The image 集成 domain was renamed to image_upload.

    This change would only affect you when you have removed the default_config from your 配置. In that case, you need to manually adjust your 配置 to rename image to image_upload.

    The rename was done to free up the image domain for future use.

    (@frenck - #84063) (文档)

    The unit for noise has been adjusted from dBa to dBA.

    (@epenet - #83581) (文档)

    AirVisual Pro units now have their own Home Assistant 集成 (as opposed to be included with the original AirVisual 集成 that uses the AirVisual cloud API).

    Pro 设备 will automatically be migrated; as part of that migration, the Pro's 设备 ID will change.

    If you utilize 实体 IDs belonging to a Pro in 自动化 or 脚本, you don't need to do anything; if, however, you utilize a Pro's 设备 ID, please 更新 those 自动化 or 脚本 accordingly.

    (@bachya - #83583) (文档)

    The previously deprecated YAML 配置 of the APC UPS Daemon 集成 has been removed.

    APC UPS Daemon is now configured via the UI. Any existing YAML 配置 has been imported in previous releases and can now be safely removed from your YAML 配置 files.

    (@yuxincs - #83801) (文档)

    The unit for sound level has been adjusted from dBa to dBA.

    (@epenet - #83582) (文档)

    An issue with the mapping of 触发器 of Styrbar has been discovered and fixed. As a result, any 自动化 relying on these 触发器 will need to be manually adjusted/re-configured to adjust to this change.

    (@Kane610 - #84257) (文档)

    Units of measurements for length, mass, time, and volume have been aligned with standard Home Assistant units.

    (@epenet - #83936) (文档)

    The unit for sound has been adjusted from dBa to dBA.

    (@epenet - #83580) (文档)

    The 更新 frequency option has been removed. The default frequency is now 60 minutes. If you like to customize the polling: disable polling 更新 from system options and use the update_entity 服务 in an 自动化 to manually 更新 it at your own custom schedule.

    (@engrbm87 - #84295) (文档)

    The previously deprecated YAML 配置 of the Tankerkoenig 集成 has been removed.

    Tankerkoenig is now configured via the UI, any existing YAML 配置 has been imported in previous releases and can now be safely removed from your YAML 配置 files.

    (@mib1185 - #84711) (文档)

    The imperial unit for Global Horizontal Irradiance has been adjusted from BTU/(h×ft²) to BTU/(h⋅ft²)

    (@epenet - #83574) (文档)

    The unit for weighted decibels has been adjusted from dBa to dBA.

    (@epenet - #83583) (文档)

    If you are a custom 集成 developer and want to learn about breaking changes and new features available for your 集成: Be sure to follow our developer 博客. The following are the most notable for this 发布:

    • Add more unit enumerators
    • Multi-pan temporarily disabled
    • Summary of unit system related changes
    • Translating the 状态 of 实体
